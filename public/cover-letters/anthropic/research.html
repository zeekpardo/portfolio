01 The Role Decoded

▼

This is not a traditional support ops role. It sits at the intersection of conversation design and support automation. You're shaping how their AI agent (Fin) communicates AND building the backend scaffolding that lets it take real action.

05 Fin Architecture & Configuration

▼

Guidance (Fin > Train > Guidance)

Natural-language instructions that coach Fin's behavior. Written in plain English, not code.

  • Communication style — Tone, voice, terminology to match brand
  • Context and clarification — When to ask follow-up questions
  • Content and sources — Which knowledge sources to prioritize
  • Escalation guidance — Nuanced, scenario-based handoff rules in natural language
  • Each rule has a channel selector (Chat, Email, Voice) for channel-specific behavior
Best Practice

Keep each rule short and single-purpose. Multiple simple rules beat one complex rule. Review for contradictions — conflicting guidance makes Fin unpredictable. Never add blanket "always offer to escalate" guidance (kills resolution rate).

Fin Attributes (Replaced AI Category Detection)

AI-detected conversation metadata. Fin automatically classifies conversations against attributes you define (issue category, urgency, sentiment, product area). No manual tagging required.

  • Power Escalation Rules (deterministic routing)
  • Drive workflow branches (route by detected intent)
  • Feed reports and analytics
  • Pass context to external systems via Data Connectors

Procedures (Replaced Fin Tasks)

Multi-step instruction flows combining natural language, code, and API calls:

Step TypePurpose
InstructionNatural language directions for Fin
Data ConnectorFetch/push data to external APIs

ConditionalDecision points and branching
CheckpointPause for human approval on high-stakes actions
Sub-procedureCall other procedures for modularity

Escalation: Two Mechanisms

Deterministic, data-basedNuanced, scenario-based

Guardrails & Safety

  • Checkpoints — Fin stops for human approval before high-stakes actions
  • Code conditions — Deterministic checks the LLM cannot override (refund eligibility, amount thresholds)
  • Defensive prompting — Protective instructions injected when LLM reads user input (prompt injection defense)
  • RAG grounding — Answers based on provided knowledge sources, not open internet
  • SOC 2 compliance — All interactions logged and auditable

06 Conversation Design

▼

Triage Architecture

Best practice: create one master Triage Workflow that fires on every new conversation (trigger: "When a customer sends their first message"), with all routing branches inside it. Fin Attributes automatically classify inbound conversations for smart routing.

Handoff Design Principles

  • Communicate that you're escalating, why, what to expect, and estimated wait times
  • Pass a full conversation summary and customer data to the human agent (warm handoff)
  • Fin generates an AI summary as an internal note at handoff
  • Use skills-based routing: billing → finance team, technical → product team

Escalation Triggers

  • Customer explicitly asks for a human
  • Fin misunderstands 2-3 times consecutively
  • Sentiment analysis detects frustration
  • Immediate-escalation keywords: "fraud," "emergency," "legal," "account locked," "outage"

The Key Failure Point

The failure point in most AI support systems is not the automation — it's the handoff. The gap between AI and human is where customer trust breaks down.

Your Direct Match

Your Grant Application Coach used the same principles: objective-based architecture where each conversation stage has one specific purpose, with automatic language detection, targeted follow-ups, and focused context per interaction to prevent hallucination.

07 Stripe & Integrations

▼

How Fin Connects to External Systems

Data Connectors — configured API calls that Fin invokes during Procedures. Each step must complete before the next (sequential, not parallel).

Stripe Integration Capabilities

  • Surface customer details: subscription plan, recent payments, billing history
  • Process refunds under custom rules (e.g., auto-refund if <$50 and within 30 days)
  • Manage subscription lifecycle: upgrades, downgrades, cancellations
  • Read-only vs. read-write access controls per action type

Example Refund Procedure

StepTypeAction
1InstructionAsk customer for order number
2Data ConnectorLook up order in Stripe
3Code ConditionCheck if within refund window (deterministic)
4ConditionalIf eligible, proceed; if not, explain policy
5CheckpointPause for manager approval if refund > $500
6Data ConnectorIssue refund via Stripe API
7InstructionConfirm refund to customer with timeline

Permission Boundaries

Define clearly what Fin can do autonomously (check payment status, look up invoices) vs. what requires human approval (large refunds, account closures, custom pricing).

10 Interview Talking Points

▼

Lead With These

  • "I've been on both sides." I've handled the tickets AND built the AI systems. Every system I build comes from real front-line friction, not theoretical problems.
  • "I think in conversation design." My Grant Coach doesn't use one giant prompt. Each conversation stage has a single objective with limited scope — that's how you prevent hallucination and maintain quality at scale.
  • "I build for adoption, not just functionality." My Spanish Support AI hit 100% team adoption because I designed it for the people closest to the problem. A tool nobody uses is worse than no tool.
  • "I understand guardrails aren't restrictions — they're what make automation trustworthy." My Grant Coach uses confidence scoring and objective-based architecture. My report builder validates Liquid syntax before delivery. Safety and speed are the same goal.

Anthropic-Specific Points

  • Mission alignment: "I wrote about Agent-First AI before seeing this role. The thesis — that AI support should have a self-improving knowledge loop — aligns directly with the Fin Flywheel's Train > Test > Deploy > Analyze cycle."
